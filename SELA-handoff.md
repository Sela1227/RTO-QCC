## 二、發現的「跨專案通用坑」（建議進 Kit）

### 提案前的檢查

已 grep V1.23.1 全部 `conventions/` 與 `SPEC.md`，確認下列三條**皆未收錄**（不是換名字，是真的沒有）：
- 坑 #14 是「PWA 版本號多檔同步」、#15 是「外部 API 單一來源」→ 主題不同
- `sela-philosophy.md:541` 有「死碼要註明刻意休眠」，那是**寫作建議**，不是「死引用靜默失效 + 死碼誤導 Claude」這條坑

### 強烈建議加坑

#### 1. 初始化函式對既有安裝早退 → 之後新增的預設值永遠補不上

- **症狀**：升版後新增的設定項，在**既有使用者**本機永遠是空的。若該設定是某必填欄位的來源 → 功能直接被擋死，且 UI 沒地方可補 → 使用者自己救不回來
- **本案事例**（V7.1.1）：`initDefaultSettings()` 開頭 `if (initialized) return;`。後來才加的 `treatment_intents`（治療目的）對舊資料庫永遠是 `[]` →「治療目的」下拉空的 → 它是新增療程的**必填欄位** → **永遠無法新增療程**，而該設定沒有管理 UI
- **關鍵證據**：該函式裡早就有人為 `patient_app_url` 單獨打補丁、註解寫「確保病人端網址有預設值（新舊用戶都適用）」—— **這坑已經咬過一次**，只補單點沒解根因
- **原因**：把「是否首次初始化」與「預設值是否齊全」混為一談。前者只該發生一次，後者每次啟動都該確保
- **做法**：預設值抽成 `DEFAULT_SETTINGS` 常數 + 每次啟動迴圈補齊**缺少的鍵**（只在 `get(key) === null` 時寫入）。三要點：
  1. 只補「完全不存在」的鍵 —— 使用者刻意清空成 `[]` 的要保留
  2. **只該跑一次的副作用（如載入演示資料）必須留在早退之後** —— 否則使用者「清除資料」後演示資料會自己長回來
  3. 之後加新設定自動補齊，不必再打單點補丁
- **影響範圍**：所有 local-first（localStorage / IndexedDB / 本機設定檔）且會持續新增設定項的 app
- **N 說明**：同專案已咬兩次（`patient_app_url`、`treatment_intents`），根因結構性、非偶發

#### 2. 純 JS 專案：inline `onclick` 死引用靜默失效，且死碼會誤導 Claude

- **症狀**：按鈕按了毫無反應，只有 console 一行 TypeError。使用者以為「功能沒做」，其實是引用打錯 / 函式被刪
- **本案事例**（V7.2.1）：使用者回報「找不到資料庫→查看→病人詳情這個鈕」。查下來每列 `onclick="App.selectPatient(id)"` —— **`App.selectPatient` 從來沒被定義過**。同次掃描再抓出兩個：`Sync.selectAndSync`（設定頁「從網芳同步」按鈕，而該功能正是專案下版第一優先、**從未能用過**）、`App.renderContent`（儲存後不刷新）
- **原因**：純 JS 無 build、無型別檢查、無 linter。HTML 字串裡的方法名是**純文字**，拼錯或函式移除都沒有建置期警告
- **⚠ 對 Claude 特別重要的一課**：本案 `js/patient.js` 有個 134 行死碼 `renderPatientList()`，寫入**與現行頁面同一個容器**，裡面有顆「查看」按鈕。Claude 讀到它就回報「這功能已經有了」—— **完全錯誤**。是使用者堅持「找不到」才揪出來
  - **通則 A：驗證 UI 必須走「渲染 → 從產出的 HTML 找按鈕 → 點它」的真實路徑**，不能直接呼叫底層函式（如 `showPatientDetail()`）就宣稱正常 —— 那會跳過 inline onclick 這層
  - **通則 B：看到疑似重複的渲染函式，先確認哪一個真的被 UI 綁定**，別照死碼推論功能
- **做法**：抓出所有 `(App|Patient|Sync|…)\.\w+\(` 呼叫，比對各物件實際定義的方法，列出差集。本案一次抓出 3 個。完整腳本見 RTO-QCC `CLAUDE.md` §5.1
- **影響範圍**：所有純 JS / 無 build / innerHTML + inline onclick 的專案（`reference-static-pages` 類全中）
- **跟 OPT-4 的關係**：OPT-4 講「生成後配驗證」的原則，本坑是它在「純 JS UI」的**具體檢查手法** —— 互補

#### 3. CDN 依賴的 PWA 在受限網路：App Shell 用 `addAll`、CDN 用 `allSettled` best-effort

- **症狀**：醫院網路封鎖某 CDN → `cache.addAll([...本地, ...CDN])` 整個 reject → SW 裝不起來 → **整個 PWA 失效**
- **原因**：`addAll` 任一失敗就全失敗；把「必須有的本地檔」和「可降級的 CDN 函式庫」綁在同一個 addAll
- **做法**：兩段式 —— 本地 App Shell `cache.addAll`（保證完整）；CDN 函式庫 `Promise.allSettled(map(cache.add))`（封鎖時降級成線上載入，SW 仍安裝成功）
- **影響範圍**：所有從 CDN 載入函式庫、又要在受限 / 離線網路跑的 PWA（醫院、政府內網）
- **跟既有坑的關係**：#13 是執行期攔截 POST、#15 是外部單點風險、#60 是可安裝性來源機制。本條是 **SW install 階段**的失敗模式，三者都沒涵蓋

### 可加但等更多證據確認

#### 列表篩選讓「存在的資料」找不到 = 使用者死路

- **共通原則**：篩選一旦讓某筆資料**存在卻搜不到**，而該筆又是使用者的唯一操作入口 → 卡死。**沒有錯誤訊息，只是「東西不見了」，比報錯更難察覺**
- **事例一（V7.1.1，把「沒有值」當成「不符合」）**：收案日期篩選 `if (!p._treatment_start) return false;`。該欄位衍生自第一筆療程，**還沒建療程的病人**沒有值 → 被當「不符合」濾掉 → 追蹤清單全以療程為基礎也不出現 → 病人被困死，永遠無法補建療程
- **事例二（V7.2.0，把「指名查找」當成「瀏覽篩選」）**：搜尋舊病人姓名 → 0 筆，因療程在往年、被「本月」擋掉。**瀏覽用篩選 vs 指名查找是兩種模式**，有搜尋字串時其他收斂條件應退讓
- **判斷：N=1 專案（同專案兩次），暫緩**。跟坑 #62 精神相近但不同層：#62 是「該不該改資料」，本條是「篩選語意」

## 三.5、發現的「程式碼優化」（建議進 optimizations.md）

### 1. OPT-2 補充：無 build / 離線優先的專案，用「常數表 + 補齊迴圈」即可，不必抽 JSON

- **現況**：OPT-2 建議「散落的初始化資料 → master JSON + seed loader」
- **本案情境**：RTO-QCC 確實中了 OPT-2 的舊寫法（種子設定散在 `initDefaultSettings()` 一長串 `Settings.set`）。V7.1.1 改成 `DEFAULT_SETTINGS` 常數 + idempotent 補齊迴圈，**達成 OPT-2 的全部目的**（單一真相、新環境一次灌好、改資料不動邏輯），但**沒有抽成 JSON 檔**
- **建議補的取捨**：
  > **不該抽 JSON 的情況**：純靜態無 build / 離線優先（PWA）的專案。抽成獨立 JSON 會多一次 `fetch`、Service Worker 要多快取一個檔、離線失敗多一個點。**程式碼內的常數表已是單一真相，收益相同、成本更低。**
  > 抽 JSON 的價值在「非工程師也能改資料」與「跨語言共用」—— 沒這兩個需求就別抽。
- **為什麼值得補**：OPT-2 現在讀起來像「JSON 才是正解」，但對 SELA 佔比很高的靜態 / 離線專案，**常數表才是正解**。少了這條取捨，下個專案的 Claude 可能為了「對齊優化庫」做一次沒必要的重構 —— 正好違反優化庫鐵律 2「無腦全套是最大風險」
- **成熟度**：[待驗證]（N=1）

## 七、給 Kit Claude 的最後備註

**本份涵蓋 6 個版本（V7.0.0～V7.3.0），因為前一份（V7.2.1 時期）的提案至今未進 V1.23.1。** 已逐條依 V1.23.1 重新 grep 查證，確認三條新坑仍是真缺口，並**主動撤回**其中一條錯誤提案。

最值得注意的三件事：

1. **「純 JS inline onclick 死引用」那條，是使用者堅持才被揪出來的。** Claude（我）先看到死碼裡的「查看」按鈕就回報「功能已存在」，錯得很徹底。這條坑的價值不只在技術，在它示範了「**Claude 拿死碼當功能證據**」這個失效模式 —— 值得讓其他專案的 Claude 看到。
2. **優化體檢（V1.17.0）確實有效。** OPT-4 直接命中上面那個失誤，促成把驗證紀律寫進專案 CLAUDE.md。這機制值得繼續投資。
3. **本份撤回了前一份的提案。**「專案 Claude 為了替既有習慣背書而提議改 Kit」是真實的偏誤風險 —— 建議 Kit Claude 收到提案時反問「這是規範缺口，還是專案習慣？」
